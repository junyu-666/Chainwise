# Chainwise 第一版设计文档

## 1. 项目定位

Chainwise 是一个面向完全没有接触过区块链用户的中文扫盲与上手网站。

它的核心价值不是再写一套区块链百科，而是提供一个可以直接转发给新手的系统入口：用生活化语言解释基础概念，用交互式模拟降低心理门槛，再用安全清单帮助用户在真实环境中避免误操作和骗局。

## 2. 一句话说明

让完全不懂区块链的人，在不被诱导交易、不暴露资产风险的前提下，理解钱包、地址、签名、转账和链上查询，并完成安全的第一次上手。

## 3. 目标用户

- 完全没接触过区块链、Web3、钱包的人
- 听过 USDT、链上转账、钱包地址，但不知道如何判断真伪和风险的人
- 对 Web3 有兴趣，但害怕被骗、怕误操作的人
- 有现实需求的人：收款、跨境支付、参与任务或空投、看懂链上记录、理解钱包安全
- 需要向朋友、社群或团队解释区块链基础的人

## 4. 产品原则

### 4.1 非投机导向

Chainwise 不做币价、行情、推荐项目、交易策略、理财收益等内容。所有内容围绕“理解基础设施”和“安全使用工具”展开。

### 4.2 分层上手：先模拟，再接真实钱包

第一版不应该只停留在模拟，也不应该让完全新手一进站就连接真实钱包。更稳妥的路径是：

1. 先用站内模拟解释概念，让用户知道钱包、地址、签名、Gas 和交易哈希分别是什么。
2. 再引导用户安装并创建真实钱包，例如 OKX 钱包。
3. 最后进入测试网任务，完成领水、测试转账、查询交易等低风险操作。

模拟钱包的价值是“认知缓冲”和“安全预演”，不是替代 OKX 钱包。它让用户在没有真实资产、没有真实助记词、没有误授权风险的情况下，先看懂即将发生的事。

### 4.3 强安全心智

安全不是附录，而是主线。每个操作模块都要解释用户正在授权什么、可能失去什么、应该检查什么。

### 4.4 生活化表达

避免一上来讲共识算法、节点、密码学细节。先用用户能理解的生活场景解释，再逐步补充必要技术概念。

### 4.5 可验证

用户不只“听懂”，还要能自己验证：看懂地址、交易哈希、确认数、区块浏览器记录、签名类型和授权对象。

### 4.6 真实钱包首选路径

第一版真实钱包实操可以优先选择 OKX 钱包，原因是它覆盖中文用户、移动端和浏览器插件场景，且新手认知成本相对低。

但产品表达上要避免变成“OKX 钱包推广页”。OKX 只是第一版示范工具，Chainwise 的核心仍然是钱包通用安全能力：助记词保护、地址识别、签名识别、测试网操作、链上查询。

后续可以扩展 MetaMask、TokenPocket、Rabby 等钱包教程，但第一版不要同时支持太多钱包，否则截图、流程、故障排查和用户心智都会变复杂。

## 5. MVP 范围

第一版只做 5 个高转化模块，不做大百科。

### 5.1 钱包到底是什么

目标：纠正常见误解。

核心内容：

- 钱包不是银行账户，也不是交易所账户
- 钱包本质上是管理私钥和发起签名的工具
- 地址像公开收款编号，私钥/助记词才是控制权
- 丢失助记词和泄露助记词分别意味着什么

交互设计：

- 模拟生成一个钱包地址
- 显示“公开信息”和“绝不能泄露的信息”的对比
- 用选择题判断哪些信息可以发给别人

### 5.2 第一次创建钱包

目标：让用户知道创建钱包时最容易犯的错。

核心内容：

- 助记词不能截图、不能发微信、不能存网盘、不能给客服
- 密码和助记词的区别
- 手机丢了、App 卸载了、换设备了分别会发生什么
- 为什么“客服要助记词”一定是骗局

交互设计：

- 新手创建钱包流程模拟
- 备份助记词风险判断
- “这句话能不能发给别人”的安全练习

### 5.3 第一次看懂地址和交易哈希

目标：让用户理解链上记录的最小单位。

核心内容：

- 地址是什么
- 交易哈希是什么
- 链不同，地址格式可能相同但资产不互通
- 转账不是截图为准，而是链上记录为准

交互设计：

- 展示一笔模拟转账
- 点击交易哈希进入模拟区块浏览器
- 标注 From、To、Amount、Token、Gas、Status、Timestamp

### 5.4 第一次模拟签名

目标：区分低风险登录签名、高风险授权和真实链上交易。

核心内容：

- 签名不是一个动作，可能代表不同含义
- message signing：常见于登录或证明地址所有权
- approve：允许某个合约使用你的代币
- permit：不花 Gas 也可能授予代币权限
- transfer / swap：会改变链上资产状态

交互设计：

- 展示 4 种签名弹窗样例
- 用户判断风险等级
- 系统解释“你正在允许谁做什么”
- 给出签名前检查清单

### 5.5 第一次查链上转账是否到账

目标：解决真实生活场景中“别人说转了，我怎么确认”的问题。

核心内容：

- 不看聊天截图，看交易哈希
- 确认链是否正确
- 确认收款地址是否是自己
- 确认代币合约是否正确
- 确认交易状态是否成功

交互设计：

- 输入或选择一笔模拟交易哈希
- 在模拟浏览器中找出到账证据
- 通过任务检查：链、状态、收款地址、金额、代币

```text
│   ├── 1. 钱包是什么
│   ├── 2. 创建钱包前必须知道的事
│   ├── 3. 地址和交易哈希
│   ├── 4. 模拟签名
│   └── 5. 查询是否到账
├── 安全中心
│   ├── 助记词安全
│   ├── 签名前检查
│   ├── 授权风险
│   └── 常见骗局
├── 场景指南
│   ├── 用稳定币收款
│   ├── 确认别人是否真的转账

```

## 7. 首页第一屏建议

首页不要做交易所式宣传，也不要堆概念。第一屏应该直接说明它是一个“安全上手入口”。

建议文案：

```text
Chainwise
给完全新手的区块链安全上手指南

先理解钱包、地址、签名和链上记录，再从模拟任务走到 OKX 钱包测试网实操。
不讲炒币，不推荐项目，不要求导入真实助记词。
```

主要行动：

- 开始 20 分钟新手路径
- 先看安全清单

第一屏下方直接露出路径进度，让用户知道接下来不是读一堆文章，而是完成几个清晰任务：

```text
1. 看懂区块链在解决什么
2. 分清钱包、地址和助记词
3. 创建 OKX 钱包
4. 连接测试网
5. 领取测试币
6. 发起一笔测试转账
7. 用区块浏览器确认结果
```

## 8. 关键交互模块

### 8.1 模拟钱包

用途：

- 生成虚拟地址
- 展示公开地址和私密凭证的区别
- 支持后续模拟签名和转账

注意：

- 必须明确“这是模拟地址”
- 不生成真实私钥
- 不要求用户连接真实钱包

### 8.2 模拟交易流水

用途：

- 解释交易从发起到确认的过程
- 展示 Pending、Success、Failed
- 解释 Gas、Nonce、Hash、Block、Confirmations

### 8.3 模拟区块浏览器

用途：

- 教用户从交易哈希验证信息
- 将真实浏览器中复杂字段简化为新手视图
- 提供“展开技术细节”入口

### 8.4 签名风险解释器

用途：

- 解析常见签名类型
- 用自然语言解释用户正在授权什么
- 输出风险等级和检查项

第一版可以先使用内置样例，不接真实钱包。

### 8.5 OKX 钱包实操引导

用途：

- 引导用户安装 OKX 钱包 App 或浏览器插件
- 创建新钱包，而不是导入已有钱包
- 解释设置密码、备份助记词、确认助记词的区别
- 引导用户切换到测试网或添加测试网络
- 连接 Chainwise 测试任务页

设计要求：

- 每一步都配“为什么要做这一步”
- 不要求用户上传、输入或截图助记词
- 页面只检查钱包连接状态、链 ID、地址和测试网余额
- 如果用户连接了主网，立即提示切换测试网，不引导真实资产操作

### 8.6 测试网任务系统

用途：

- 领取测试币
- 发起一笔测试转账
- 等待交易确认
- 打开区块浏览器查询交易哈希
- 回到 Chainwise 标记任务完成

任务状态：

- 未开始
- 已连接钱包
- 网络正确
- 已领取测试币
- 已发起测试转账
- 已确认到账
- 已完成复盘

第一版可以优先选择一个 EVM 测试网，例如 Base Sepolia 或 Sepolia。最终选择需要结合领水稳定性、浏览器易用性、OKX 钱包支持情况和国内访问体验决定。

## 9. 首次用户路径

这一节描述用户第一次进入网站后实际看到什么、点击什么、学到什么。

### 9.1 首页

用户看到：

- 标题：给完全新手的区块链安全上手指南
- 副标题：从生活化解释到 OKX 钱包测试网实操
- 主按钮：开始 20 分钟新手路径
- 次按钮：先看安全清单
- 路径条：理解概念 -> 创建钱包 -> 连接测试网 -> 领水 -> 测试转账 -> 查询结果

首页首屏不放行情、不放热门项目、不放收益故事，避免用户把产品误解为交易入口。

### 9.2 第一步：连接区块链概念

用户进入后先看到一个生活化问题：

```text
如果有人说已经给你转了一笔钱，你怎么确认这件事是真的？
```

然后用三层解释连接区块链概念：

- 普通转账依赖银行或平台记录
- 区块链转账依赖公开账本记录
- 钱包不是账本，钱包只是帮你证明“这个地址由你控制”

这一页只要求用户理解三个词：

- 地址：别人可以看到、可以给你转账的公开编号
- 签名：你证明自己同意某个操作
- 交易哈希：一笔链上操作的查询编号

完成条件：

- 用户能回答“钱包里是否真的存着币”
- 用户能回答“为什么截图不能证明到账”

### 9.3 第二步：模拟钱包

系统生成一个模拟地址，例如：

```text
0x8a3...91F2
```

页面把信息分成两列：

- 可以公开：地址、交易哈希
- 绝不能泄露：助记词、私钥、验证码、远程控制权限

这一页的作用是让用户先分清“公开身份”和“控制权”。如果直接跳到 OKX 创建钱包，新手很容易把密码、助记词、地址混在一起。

完成条件：

- 用户能判断“地址可以发给别人”
- 用户能判断“助记词不能截图、不能发给客服”

### 9.4 第三步：创建 OKX 钱包

页面引导用户选择设备：

- 手机 App
- 浏览器插件

第一版建议默认引导手机 App，因为新手更熟悉移动端流程；浏览器插件作为可选路径。

关键步骤：

1. 下载并打开 OKX 钱包
2. 选择创建新钱包
3. 设置本地密码
4. 备份助记词
5. 完成助记词确认
6. 回到 Chainwise

每一步都要加安全提示：

- 本地密码不是助记词
- 助记词不能截图
- Chainwise 不会要求你输入助记词
- 任何人索要助记词都应立即停止

完成条件：

- 用户确认已创建新钱包
- 用户确认已离线备份助记词
- 用户确认没有把助记词输入 Chainwise

### 9.5 第四步：连接钱包

用户点击“连接测试任务”，选择 OKX 钱包。

连接后页面展示：

- 当前地址
- 当前网络
- 是否为测试网
- 当前测试币余额

如果用户在主网：

```text
你现在连接的是主网。为了避免真实资产风险，请切换到测试网后继续。
```

完成条件：

- 钱包已连接
- 网络为指定测试网
- 页面能读取测试网地址和余额

### 9.6 第五步：测试网领水

页面解释：

- 测试币没有真实价值
- 测试币用于支付测试交易 Gas
- 领水不是空投，不需要授权代币

用户点击“领取测试币”后，页面显示：

- 领水入口
- 当前地址
- 预计到账时间
- 到账后余额变化

如果领水失败，给出替代方案：

- 换一个水龙头
- 稍后再试
- 使用站内演示交易继续学习

完成条件：

- 测试币余额大于 0

### 9.7 第六步：测试转账

页面提供一个 Chainwise 控制的测试收款地址，或让用户向自己的第二个测试地址转账。

转账前检查：

- 网络是否正确
- 收款地址是否正确
- 金额是否为测试币
- Gas 是否为测试币支付
- 是否没有授权任何代币

用户在 OKX 钱包中确认后，Chainwise 显示交易状态：

- Pending：等待打包
- Success：交易成功
- Failed：交易失败，解释可能原因

完成条件：

- 页面拿到交易哈希
- 交易最终状态为 Success 或用户理解失败原因

### 9.8 第七步：区块浏览器确认

用户点击交易哈希进入区块浏览器，或者在站内简化浏览器视图查看。

需要用户找到：

- Status 是否成功
- From 是否是自己的地址
- To 是否是收款地址
- Value 是否是测试金额
- Transaction Hash 是否匹配

完成条件：

- 用户能独立判断这笔测试转账是否成功

### 9.9 第八步：复盘

最后一页不讲更多概念，只复盘用户已经完成的真实能力：

- 我知道钱包不是账户
- 我知道地址可以公开，助记词绝不能泄露
- 我知道连接钱包不等于转账
- 我知道签名和授权要分开看
- 我知道如何用交易哈希确认到账
- 我完成过一次测试网转账

这一页可以生成学习完成页，后续再扩展证书或分享卡。

## 10. 登录与学习进度

第一版建议做登录和进度，但不要一开始做重账号体系。

### 10.1 为什么需要进度

进度模块的价值很明确：

- 记住用户学到哪里
- 记录用户完成了哪些安全题
- 记录用户是否完成测试网连接、领水、转账、查询
- 后续扩展证书、任务、复训和企业培训

### 10.2 第一版推荐方案

采用三层进度能力：

1. 本地进度：未登录用户也能用浏览器 localStorage 保存学习进度。
2. 钱包进度：连接钱包后，用钱包地址绑定测试网任务进度。
3. 账号进度：后续再支持手机号、邮箱或第三方登录，用于跨设备同步。

第一版优先实现前两层：本地进度 + 钱包地址进度。

### 10.3 钱包地址是否等于用户账号

钱包地址可以作为学习身份的一部分，但不应该直接等同于完整用户账号。

原因：

- 用户可能有多个钱包地址
- 用户可能换设备或换钱包
- 钱包地址是公开身份，不适合承载所有个人资料
- 新手可能还没理解地址公开性的含义

更合适的设计是：

- 未连接钱包时，用本地匿名进度
- 连接钱包后，把测试网任务进度绑定到地址
- 当用户需要跨设备、证书或长期学习记录时，再引导创建账号

### 10.4 登录方式优先级

第一版：

- 不强制登录
- 允许直接开始学习
- 在“连接钱包实操”前才提示连接 OKX 钱包
- 连接钱包只用于读取地址、网络和测试任务状态

第二版：

- 支持 Sign-In with Ethereum 类似的钱包登录
- 支持邮箱登录同步学习进度
- 支持把多个钱包地址绑定到同一个学习账号

### 10.5 数据模型草案

```text
UserProgress
- anonymousId
- accountId
- walletAddress
- chainId
- currentStep
- completedLessons
- quizResults
- testnetTasks
- lastSeenAt

TestnetTask
- walletAddress
- chainId
- faucetClaimed
- faucetTxHash
- transferTxHash
- transferStatus
- explorerChecked
- completedAt
```

隐私原则：

- 不收集助记词、私钥、真实资产余额
- 不默认展示用户完整地址
- 不把测试网行为包装成投资成就
- 删除账号时应能删除学习进度

## 11. 安全内容边界

必须明确避免：

- 投资建议
- 买币教程
- 收益承诺
- 项目推荐
- 空投推荐
- 引导用户连接陌生 dApp
- 引导用户导入真实助记词

必须反复强调：

- 任何人索要助记词都是风险
- 任何客服、老师、群友都不应该看到助记词
- 真实转账前先确认链、地址、代币和金额
- 授权不是登录，授权可能导致资产被转走

## 12. 常见骗局模块

第一版至少覆盖：

- 假空投：要求连接钱包并授权代币
- 假客服：要求提供助记词、私钥或远程控制
- 授权盗币：诱导 approve 或 permit
- 剪贴板木马：复制地址后被替换
- 假充值截图：用截图冒充链上转账
- 同名假币：代币名称相同但合约地址不同
- 错链转账：资产发到了错误网络

## 13. 内容语气

语气要求：

- 简明、克制、可信
- 不恐吓，但明确风险
- 不把用户当技术人员
- 不使用币圈黑话作为默认表达

示例：

```text
不准确：钱包里存着你的币。
更准确：钱包帮你管理控制权。链上的资产记录不在钱包 App 里，而在区块链上。
```

```text
不准确：签名一下就能登录。
更准确：有些签名只是证明“这个地址属于你”，有些签名会授权别人动用你的资产。签名前要看清类型。
```

## 15. 后续产品方向

### 15.1 安全检查工具

用户粘贴签名内容、交易信息或授权内容，系统解释风险等级。

### 15.2 测试网上手任务

引导用户使用测试网完成创建地址、领取测试币、发送交易、查询交易、理解失败交易。

### 15.3 钱包新手陪练

在严格安全提示下，让用户连接测试网钱包完成 5 个任务，并获得链上入门证书。

### 15.4 企业或社群培训版

为传统团队、社群、新用户 onboarding 提供标准化课程和进度追踪。

## 16. 成功指标

第一版可以关注：

- 新手路径完成率
- 每个交互任务完成率
- 安全题正确率
- 用户是否能独立判断“是否到账”
- 用户是否能区分登录签名和授权签名
- 文档或页面被转发次数
- 搜索流量入口词：区块链是什么、钱包是什么、USDT 怎么确认到账、交易哈希是什么

## 17. 第一阶段实施建议

建议优先顺序：

1. 搭建静态内容站和基础视觉风格
2. 完成首页与 5 个概念 MVP 模块
3. 做模拟钱包、模拟交易、模拟浏览器三个轻交互组件
4. 增加 OKX 钱包创建引导和测试网连接任务
5. 增加测试网领水、测试转账、区块浏览器确认流程
6. 加入本地进度和钱包地址进度
7. 加入安全中心和常见骗局内容
8. 再考虑真实资产、主网操作、链上证书和完整账号体系

第一版要保持克制：可以接真实钱包，但只做测试网；可以记录进度，但不强制登录；可以教用户查真实区块浏览器，但不引导真实资产操作。
